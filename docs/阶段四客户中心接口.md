# 阶段四客户中心接口

## 数据库版本与部署状态

Flyway `V11__create_customer_project_applications.sql` 新增
`project_customer_application`，保存客户与婚礼项目的关联申请：

- 同一客户与同一项目只保留一条逻辑记录。
- 状态包括 `PENDING`、`APPROVED`、`REJECTED`。
- 已驳回申请使用当前 `version` 重新提交，不创建重复记录。
- 审核人、审核时间、驳回原因和逻辑删除字段均保留。

Flyway `V12__create_customer_notifications.sql` 最初新增 `customer_notification`，保存客户
站内消息。Flyway `V14__generalize_user_notifications.sql` 已将其重命名为
`user_notification`，供客户、管理员和创作者共用：

- 消息按客户收件人隔离，记录消息类型、标题、内容、关联业务、阅读时间和乐观锁版本。
- 客户消息类型包括项目关联通过或驳回、客户评价通过或驳回、客户评价下架。
- 消息保留审计字段和逻辑删除字段，单条已读递增版本，全部已读批量更新阅读时间和版本。
- 消息内容不会写入完全隐藏项目的标题、日期、地点或新人称呼，项目关联消息只使用项目编号。
- `V14` 同时新增 `/notifications` 权限资源，并分配给 `ADMIN` 和 `CREATOR`。

客户账号、`CUSTOMER` 系统角色、`customer_profile` 和 `/accounts/customers` 权限资源已由
Flyway `V1` 至 `V3` 建立，因此本次不重复创建。

`V11`、`V12`、`V14` 已于 2026-07-18 完成生产部署，生产 schema 当前为 `V14`。
生产表为 `user_notification`，旧表 `customer_notification` 已不存在。

## 客户注册与账号

### `POST /api/auth/register`

公开注册客户账号，注册成功后直接返回 JWT 和当前客户账号：

```json
{
  "mobile": "13800000000",
  "password": "customer-password",
  "nickname": "林女士"
}
```

规则：

- 手机号必须为 11 位中国大陆手机号。
- 密码长度为 8 至 72 个字符，使用 BCrypt 摘要保存。
- 昵称最多 100 个字符。
- 手机号已被任意类型账号使用时返回 `409 MOBILE_EXISTS`。
- 新账号分配固定 `CUSTOMER` 系统角色，不分配工作台权限。
- 第一阶段不发送短信验证码。

### 共用账号接口

客户与管理员、创作者共用：

- `POST /api/auth/login`
- `GET /api/auth/me`
- `PUT /api/account/password`

客户专用资料接口：

#### `PUT /api/account/customer-profile`

```json
{
  "nickname": "林女士"
}
```

只允许 `CUSTOMER` 账号修改，保存后同步更新 `sys_user.display_name` 和
`customer_profile.nickname`。

### 管理员客户账号接口

以下接口要求管理员角色和权限资源 `/accounts/customers`：

- `GET /api/admin/customers`：查询全部未删除客户账号。
- `PATCH /api/admin/customers/{customerId}/status`：启用或停用客户账号。
- `POST /api/admin/customers/{customerId}/reset-password`：人工核验后设置临时密码。

状态请求：

```json
{
  "status": "DISABLED"
}
```

重置密码请求：

```json
{
  "initialPassword": "temporary-password"
}
```

重置后 `mustChangePassword=true`。客户下次登录后只能访问当前账号和修改密码接口，完成改密后
才恢复项目关联、评价和资料修改。停用账号后，当前 JWT 和后续密码登录都会被拒绝。

## 客户项目关联

客户接口统一要求 `CUSTOMER` 角色。

### `GET /api/customer/projects/applications`

返回当前客户全部未删除关联申请，按创建时间倒序排列。

### `GET /api/customer/projects/linked`

只返回当前客户状态为 `APPROVED` 的项目关联。

### `POST /api/customer/projects/applications`

首次提交：

```json
{
  "version": null,
  "projectCode": "TS20260718",
  "applyNote": "我是该场婚礼的新娘，婚礼日期为 2026-07-18"
}
```

已驳回后重新提交：

```json
{
  "version": 2,
  "projectCode": "TS20260718",
  "applyNote": "补充登记手机号和新人称呼"
}
```

规则：

- 项目编号只允许字母、数字、短横线和下划线，长度为 4 至 32。
- 申请说明必填，最多 1000 个字符。
- 已存在 `PENDING` 申请时返回冲突。
- 已存在 `APPROVED` 关联时返回冲突。
- 重新提交 `REJECTED` 申请必须携带当前 `version`。
- 完全隐藏或未公开项目允许建立关联，供客户评价项目参与创作者。
- 客户响应不会返回完全隐藏或已删除项目的标题、新人称呼、日期和地点。

### 管理员审核接口

以下接口要求管理员角色和权限资源 `/accounts/customers`：

- `GET /api/admin/customer-project-applications`
- `POST /api/admin/customer-project-applications/{applicationId}/approve`
- `POST /api/admin/customer-project-applications/{applicationId}/reject`

列表参数：

- `page`：从 0 开始。
- `size`：1 至 100。
- `status`：可选 `PENDING`、`APPROVED`、`REJECTED`。

通过请求：

```json
{
  "version": 0
}
```

驳回请求：

```json
{
  "version": 0,
  "reason": "申请说明不足，请补充婚礼日期和新人称呼"
}
```

只有 `PENDING` 申请可以审核，审核使用乐观锁。项目已删除时不能通过申请。

## 客户直接评价

客户接口统一要求 `CUSTOMER` 角色。

### `GET /api/customer/feedback`

查询当前客户自己提交的评价，支持：

- `page`、`size`
- `reviewStatus`
- `publishStatus`

### `GET /api/customer/feedback/options`

返回当前客户已审核关联且仍未删除的项目，以及这些项目中的有效参与创作者。

完全隐藏项目保留项目编号和参与创作者选项，但不返回项目标题。

### `POST /api/customer/feedback`

```json
{
  "projectId": 12,
  "creatorUserId": 8,
  "rating": 5,
  "content": "记录了很多真诚瞬间"
}
```

### `PUT /api/customer/feedback/{feedbackId}`

修改自己尚未公开的评价，并携带当前 `version`：

```json
{
  "version": 1,
  "projectId": 12,
  "creatorUserId": 8,
  "rating": 5,
  "content": "补充后的评价内容"
}
```

保存后清除旧驳回结论，重新进入 `PENDING + UNPUBLISHED`。

### `DELETE /api/customer/feedback/{feedbackId}?version={version}`

撤回自己尚未公开的评价，成功返回 `204`。

客户评价规则：

- 所选项目必须存在当前客户的 `APPROVED` 关联。
- 被评价账号必须是该项目创建者或参与创作者。
- 客户称呼从当前客户昵称派生，不信任请求传入的客户姓名。
- 同一客户可以对同一创作者提交多条评价。
- 公开或下架后评价锁定，客户不能修改或撤回。
- 管理员继续使用阶段四评价审核接口完成通过、驳回和下架。

## 客户站内消息

客户消息接口统一要求 `CUSTOMER` 角色，且服务端按当前 JWT 中的客户编号限制收件人范围。

### `GET /api/customer/notifications`

查询当前客户的消息，参数：

- `page`：从 0 开始。
- `size`：1 至 100。
- `unreadOnly`：可选，默认 `false`；为 `true` 时只返回未读消息。

分页响应除标准分页字段外，同时返回当前客户的总未读数 `unreadCount`。每条消息包含：

- `type`：消息类型。
- `title`、`content`：展示内容。
- `relatedType`、`relatedId`：关联的项目申请或评价记录。
- `readAt`：为空表示未读。
- `createdAt`、`version`。

### `GET /api/customer/notifications/unread-count`

返回当前客户未读数量：

```json
{
  "unreadCount": 3
}
```

### `POST /api/customer/notifications/{notificationId}/read`

单条标记已读，必须携带当前版本：

```json
{
  "version": 0
}
```

只能操作当前客户自己的消息。消息不存在或属于其他客户时统一返回
`404 USER_NOTIFICATION_NOT_FOUND`，版本不一致时返回
`409 USER_NOTIFICATION_VERSION_CONFLICT`。

### `POST /api/customer/notifications/read-all`

将当前客户全部未读消息批量标记为已读，返回本次更新数量和统一阅读时间：

```json
{
  "updatedCount": 3,
  "readAt": "2026-07-18T12:00:00Z"
}
```

消息生成规则：

- 项目关联申请通过或驳回后，向申请客户生成消息。
- 只有客户本人通过客户中心提交、且 `customer_user_id` 非空的评价才生成审核和下架消息。
- 管理员或创作者代提交的评价没有客户账号收件人，不生成客户消息。
- 消息跳转由 `relatedType` 决定：项目申请进入项目关联页，评价进入我的评价页。

## 工作台站内消息

工作台消息接口要求 `ADMIN` 或 `CREATOR` 角色，并同时要求 `/notifications` 权限：

- `GET /api/notifications`
- `GET /api/notifications/unread-count`
- `POST /api/notifications/{notificationId}/read`
- `POST /api/notifications/read-all`

分页、只看未读、响应结构、单条已读版本校验和全部已读行为与客户消息接口一致。消息严格按
当前 JWT 对应账号隔离，管理员、创作者和客户不能读取或修改其他收件人的消息。

工作台消息覆盖：

- 项目和作品集提交审核，以及审核通过或驳回结果。
- 项目和作品集参与创作者新增或移除。
- 项目和作品集发布或下架。
- 客户项目关联申请、客户评价提交和重新提交。
- 客户评价审核、评价回复提交，以及回复通过或驳回。
- 官网新增咨询线索。

管理员广播只发送给当前有效管理员，参与关系和内容结果只发送给当前有效创作者；触发操作的
账号不会收到自己产生的同一条广播消息。

## 前端入口

官网：

- `/customer/login`
- `/customer/register`
- `/customer/messages`
- `/customer/projects`
- `/customer/feedback`
- `/customer/settings`

官网客户认证使用独立的本地存储键，不与 `/console/` 工作台认证共享。首页顶部工具区新增客户
中心入口。

运营工作台：

- `/console/customers`
- `/console/notifications`

页面按“客户账号”和“项目关联审核”分区，入口同时受前端权限和后端方法级权限控制。客户账号
不能登录运营工作台。工作台侧边栏、桌面工具区和移动导航显示站内消息入口及未读状态。
